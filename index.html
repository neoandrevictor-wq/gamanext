O "Prompt Alma"

A base lógica que injetamos na IA para criar sistemas ultrarrápidos, polidos e sem dependências.

gama_sys_architect_v3.md
Aja como um Engenheiro de Software Sênior especialista em Vanilla JS, Algoritmos Complexos e Web APIs nativas (Canvas, WebGL, FileReader, WebRTC). Sua missão é criar uma ferramenta profissional, polida e de nível comercial em um ÚNICO ARQUIVO HTML (contendo CSS via Tailwind CDN e JS inline). A ferramenta será: [INSERIR DESCRIÇÃO DO SISTEMA/VISUALIZADOR/EDITOR AQUI]. Independentemente da complexidade solicitada, o código final DEVE OBRIGATORIAMENTE seguir a seguinte arquitetura: 1. REGRA DE OURO (ZERO ASSETS EXTERNOS): Não use imagens externas, backend, arquivos de áudio ou frameworks pesados como React/Vue. Toda a interface deve ser renderizada via Tailwind CSS e toda a lógica matemática/dados rodará no lado do cliente (Client-Side Processing). 2. PERFORMANCE & ARQUITETURA: - Utilize Classes ES6 para gerenciar as "Engines" do sistema. - Se for processamento de imagem: Acesse os dados bruto via `ctx.getImageData`, opere os canais RGBA com Uint8ClampedArray e evite loops aninhados lentos. - Gerencie Estados de UI de forma clara (Carregando, Processando, Finalizado). 3. DESIGN SYSTEM (CYBER/MODERN): - Fundo muito escuro (#050505), fontes modernas (JetBrains Mono e Rajdhani/Inter). - Bordas sutis, glassmorphism sutil (backdrop-blur) e acentos de cores vibrantes. - Feedbacks visuais intensos para o usuário (Loaders de console, badges piscando, logs simulados). 4. TRATAMENTO DE ERROS & UX: - Valide inputs (ex: Checar se o arquivo é imagem, checar limites de memória para resoluções gigantes). - Se a operação for pesada, utilize `setTimeout` ou `requestAnimationFrame` para não "congelar" a UI (permitindo que loaders rodem enquanto o processamento JS ocorre em background). Escreva o código fonte HTML completo, comentado e pronto para produção em ambiente real.
⚠️ Pílula de Conhecimento: Limite de Contexto e O Método de Partição

Sistemas como o *Gama Repainter* ou *Obliterator* contêm matemática complexa (Cálculos de distância Euclidiana de cor, Convolução de Matrizes, etc). Se você pedir para a IA criar tudo em um único prompt de uma vez, ela omitirá detalhes ou criará bugs de performance.

O Pipeline de Construção:

  1. Use o prompt acima definindo a casca visual e a infraestrutura HTML/Tailwind primeiro.
  2. Teste a UI. Se estiver perfeita, envie o segundo comando: "A UI está ótima. Agora vamos focar APENAS na função matemática de extração do histograma. Crie o algoritmo para ler os pixels com performance."
  3. Avance módulo por módulo (Upload -> Preview -> Algoritmo Core -> Exportação).
  4. Peça micro-otimizações: "Esse loop for duplo está travando a tela. Reescreva utilizando Web Workers ou dividindo em chunks no RequestAnimationFrame."

O Conceito: Nano Frontend

Por que eu parei de construir grandes aplicações e comecei a criar produtos de uma página

Durante muito tempo, o “caminho certo” do frontend parecia claro: SPAs enormes, arquiteturas complexas, múltiplas camadas, backend obrigatório, pipelines longos.


Mas algo mudou.


Com a chegada da IA como força real de produção de código, ficou evidente para mim que o gargalo não é mais escrever código, e sim definir bem o escopo, entregar rápido e provar valor. Foi daí que nasceu a abordagem arquitetural da Gama Next.

O que é?

Uma abordagem onde cada página é um produto completo. Em vez de um “grande app”, você tem vários pequenos produtos autônomos.

  • Um único arquivo HTML contém tudo.
  • Não há backend ou dependências externas.
  • Não há build step (Webpack/Vite).
  • Cada deploy é 100% independente.
  • Se precisar evoluir, evolui. Se descartar, descarta.

Por que hoje?

Porque a realidade do mercado mudou. A velocidade de entrega virou um diferencial real contra a paralisia de análise.

  • IA acelera drasticamente a criação lógica.
  • O custo de experimentar ideias caiu a quase zero.
  • A web voltou a valorizar a simplicidade extrema.
  • O portfólio virou mais importante que a promessa.
  • O foco sai da "arquitetura teórica" para o resultado.

Não é Micro-Frontend

Micro-frontends nasceram para resolver problemas de coordenação em grandes empresas e times gigantescos.

  • Nano Frontend nasce para desenvolvedores solo.
  • Ideal para times ágeis, freelancers e MVPs rápidos.
  • Focado em demonstrações técnicas reais.
  • Sem overhead de orquestração de rotas.
  • Desacoplamento absoluto e brutal.

"Você não mostra prints. Você mostra links."

Cada Nano Frontend é, ao mesmo tempo, um produto funcional, um experimento técnico, um case de arquitetura e uma peça viva de portfólio.

Onde estou aplicando isso?

Na própria Gama Next. Todos os painéis, suítes de imagem e jogos acima são Nano Frontends focados em dados e processamento client-side, rápidos e autônomos.


Nano Frontend não é sobre fazer menos.
É sobre fazer o suficiente, bem feito, e entregar rápido.
